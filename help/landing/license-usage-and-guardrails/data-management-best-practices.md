# データ管理ライセンス権限のベストプラクティス

Adobe Experience Platformは、データをリアルタイムで更新する堅牢な顧客プロファイルに変換するオープンシステムで、AI に基づくインサイトを使用して、あらゆるチャネルにわたって適切なエクスペリエンスを提供します。 ソースを使用して、様々なタイプ、ボリューム、履歴のデータをExperience Platformに入力し、そのデータを、セグメント化やパーソナライゼーション、分析、機械学習などの使用例に注意できます。

Platform は、作成可能なプロファイルの数と取り込み可能なデータ量を確立するライセンスを提供します。 データのソース、ボリューム、履歴を取り込む能力を考慮すると、データ量の増加に伴い、ライセンス権限を超えることが可能です。

このドキュメントでは、Adobe Experience Platformでのライセンス使用権限をより適切に管理するために使用できる、従うベストプラクティスとツールの概要を説明します。

## Adobe Experience Platformデータストレージについて

Experience Platformは、主に次の 2 つのデータリポジトリで構成されます。の [!DNL Data Lake] とプロファイルストア。

この **[!DNL Data Lake]** 主に次の目的に使用します。

* データをExperience Platformにオンボーディングするステージング領域として機能する
* すべてのExperience Platform・データの長期的なデータ・ストレージとして機能
* データ分析やデータサイエンスなどの使用例を有効にします。

この **プロファイルストア** は、顧客プロファイルを作成する場所で、主に次の目的を果たします。

* リアルタイムエクスペリエンスのサポートに使用されるプロファイルのデータストレージとしての役割を果たします。
* セグメント化、アクティベーション、パーソナライゼーションなどの使用例を有効にします。

>[!NOTE]
>
>へのアクセス [!DNL Data Lake] は、購入した製品 SKU に応じて異なります。 製品 SKU の詳細については、Adobe担当者にお問い合わせください。

## ライセンス使用状況 {#license-usage}

ライセンスExperience Platformには、SKU によって異なるライセンス使用権限が付与されます。

**[!DNL Addressable Audience]**  — 既知のプロファイルと偽名プロファイルの両方を含む、契約上Experience Platformで許可された顧客プロファイルの合計数。

**[!DNL Profile Richness]**  — プロファイルデータのExperience Platform平均サイズ。 次の項目を増やすと、 [!DNL Profile Richness] 豊富なパックを購入することによる資格。

この [!DNL Profile Richness] 指標は、購入したライセンスによって異なります。 次の 2 つの計算があります。 [!DNL Profile Richness] 利用可能：

* 任意の時点でReal-time Customer Data Platform（プロファイルサービスおよび ID サービス）に保存されるすべての実稼動データの合計を、 [!DNL Addressable Audience];
* Platform に保存されるすべてのデータの合計 ( [!DNL Data Lake]、プロファイルサービス、ID サービス ) を任意の時点で、および（過去 12 ヶ月間に）Platform 内で（保存するのではなく）ストリーミングするデータを、 [!DNL Addressable Audience].

これらの指標の使用可否と各指標の定義は、組織が購入したライセンスによって異なります。

## ライセンス使用状況ダッシュボード

Adobe Experience Platform UI には、組織の Platform のライセンス関連データのスナップショットを表示できるダッシュボードが用意されています。 スナップショットが作成された特定の時点で表示されたとおりに、ダッシュボードのデータが表示されます。 スナップショットはデータの近似でもサンプルでもなく、ダッシュボードはリアルタイムで更新されません。

詳しくは、 [Platform UI でのライセンス使用状況ダッシュボードの使用](../../dashboards/guides/license-usage.md#license-usage-dashboard-data).

## データ管理のベストプラクティス

以下の節では、データをより適切に管理するためのベストプラクティスの概要を説明します。

### データについて

すべてのデータがAdobe Experience Platformで同じではありません。 データの中には密で値が低いものもあれば、疎で値が高いものもあります。 データによっては、生成後すぐに価値が失われる場合がありますが、年ではなく数か月間、価値がある場合もあります。

データの価値を理解する際に考慮すべき次元は 3 つあります。

| ディメンション | 説明 | 例 |
| --- | --- | --- |
| ボリューム | 取り込まれるデータの量と合計を表します。 | Web クリック — ボリュームが多く、忠実性が適度に高い。 値は急速に減少する可能性があります。 |
| 期間 | 取り込まれたデータが引き続き価値を保つ時間の長さを表します。 | オフライン購入 — ボリュームと正確性は適度に維持されますが、長期間にわたって役立つ場合があります。 |
| 正確さ | データの情報量がどれだけ多いかを表します。 | 顧客アカウント：ボリュームは少ないが、正確性は高い。 顧客の生涯を超えて価値を提供 |

### データ管理ツール {#data-management-tools}

データの使用状況をライセンス使用権限の制限内に保つ場合は、主に次の 2 つのシナリオを考慮する必要があります。

### Platform に取り込むデータ

データは、Platform の 1 つ以上のシステム ( [!DNL Data Lake] プロファイルストア つまり、様々な使用例に対して、両方のシステムに異なるデータが存在する可能性があります。 例えば、履歴データを [!DNL Data Lake]ですが、プロファイルストアにはありません。 プロファイル取り込み用のデータセットを有効にすることで、プロファイルストアに送信するデータを選択できます。

>[!NOTE]
>
>へのアクセス [!DNL Data Lake] は、購入した製品 SKU に応じて異なります。 製品 SKU の詳細については、Adobe担当者にお問い合わせください。

### 保持するデータ

データ取り込みフィルターと有効期限ルール（有効期間「TTL」とも呼ばれます）の両方を適用して、使用事例で古くなったデータを削除できます。 通常、行動データ（Analytics データなど）は、レコードデータ（CRM データなど）よりも大幅にストレージを消費します。 例えば、多くの Platform ユーザーは、レコードデータに比べて、行動データのみで入力されるプロファイルの最大 90%を持っています。 したがって、ライセンス権限内のコンプライアンスを確保するには、行動データの管理が重要です。

ライセンス使用権限を維持するために利用できるツールは多数あります。

* [取り込みフィルター](#ingestion-filters)
* [プロファイルサービス TTL](#profile-service)

### プロファイルサービス {#profile-service}

「プロファイルサービスの TTL（有効期限）」機能を使用すると、プロファイルストアのデータに TTL を適用できます。 これにより、時間の経過と共に価値が低下したデータを自動的に削除できます。

プロファイルストアは、次のコンポーネントで構成されています。

| プロファイルストアコンポーネント | 説明 |
| --- | --- |
| プロファイルフラグメント | 各顧客プロファイルは、複数の **プロファイルフラグメント** を統合して、その顧客の単一のビューを形成している。 例えば、顧客が複数のチャネルをまたいでブランドとやり取りする場合、組織は複数のチャネルを持ちます **プロファイルフラグメント** 複数のデータセットに表示される単一の顧客に関連する これらのフラグメントが Platform に取り込まれると、それらは ID グラフを使用して結合され、その顧客の単一のプロファイルが作成されます。 **プロファイルフラグメント** は、識別子として id 名前空間と、関連するレコードデータや時系列データで構成されます。 |
| レコードデータ（属性） | プロファイルとは、多数の **属性** ( 別名 **レコードデータ**) をクリックします。 例えば、製品のプロファイルには SKU と説明が含まれ、人のプロファイルには名、姓、電子メールアドレスなどの情報が含まれる場合があります。**レコードデータ** は通常、音量が低いか中程度ですが、長期間有効です。 |
| 時系列データ（動作） | **時系列データ** は、ユーザーの行動に関する情報を提供します。 標準スキーマクラスのエクスペリエンスデータモデル (XDM) で表されます。 [!DNL ExperienceEvent]の時系列データは、買い物かごに追加された項目、クリックされたリンク、ビデオの閲覧などのイベントを記述できます。 行動の値は、時間の経過と共に減少する場合があります。 |
| ID 名前空間 (ID) | 顧客データが統合されると、 **ID 名前空間**&#x200B;を使用すると、ユーザーに関するより多くの情報が得られるので、ID を結び付ける機能が得られます。 詳しくは、 [ID 名前空間の概要](../../identity-service/namespaces.md) を参照してください。 |

#### プロファイルストア構成レポート

プロファイルストアの構成を理解するのに役立つレポートが多数用意されています。 これらのレポートは、プロファイルの TTL を設定して、ライセンスの使用状況を最適化する方法と場所に関する十分な情報に基づく決定をおこなうのに役立ちます。

* **データセット重複レポート API**:アドレス可能なオーディエンスに最も貢献するデータセットを公開します。 このレポートを使用して、どのレポートを特定できるかを確認できます [!DNL ExperienceEvent] の TTL を設定するデータセット。 に関するチュートリアルを参照してください。 [データセットの重複レポートの生成](../../profile/tutorials/dataset-overlap-report.md) を参照してください。
* **ID 重複レポート API**:アドレス可能なオーディエンスに最も貢献する ID 名前空間を公開します。 に関するチュートリアルを参照してください。 [id 重複レポートの生成](../../profile/api/preview-sample-status.md#generate-the-identity-namespace-overlap-report) を参照してください。
* **不明なプロファイルレポート API**:異なる時間しきい値に偽名の TTL を適用する影響を公開します。 このレポートを使用して、適用する偽名の TTL しきい値を識別できます。 に関するチュートリアルを参照してください。 [不明なプロファイルレポートの生成](../../profile/api/preview-sample-status.md#generate-the-unknown-profiles-report) を参照してください。

#### [!DNL ExperienceEvent] データセットの TTL {#dataset-ttl}

TTL をプロファイル対応データセットに適用して、使用事例にとって役立たなくなった行動データをプロファイルストアから削除できます。 プロファイルが有効なデータセットに TTL が適用されると、Platform は、次の 2 つの手順で不要になったデータを自動的に削除します。

* 今後のすべての新しいデータの取り込み時に、TTL 有効期限値が適用されます。
* 既存のすべてのデータには、1 回限りのバックフィルシステムジョブの一環として TTL 有効期限値が適用されます。

各イベントの TTL 値は、イベントタイムスタンプから取得されます。 TTL 有効期限値より古いイベントは、システムジョブの実行時に即座に破棄されます。 その他のイベントは、イベントタイムスタンプで指定された TTL 有効期限値に近づくとドロップオフされます。

次の例を参照して、 [!DNL ExperienceEvent] データセットの TTL。

5 月 15 日に 30 日の TTL 値を適用した場合は、次のようになります。

* すべての新しいイベントの TTL は、受信時に 30 日間適用されます。
* 4 月 15 日より前のタイムスタンプを持つ既存のイベントはすべて、システムジョブによって直ちに削除されます。。
* 4 月 15 日以降にタイムスタンプを持つイベントは、イベントタイムスタンプ+ TTL 日の有効期限が切れます。 したがって、4 月 18 日のタイムスタンプを持つイベントは、5 月 15 日から 3 日後に減少します。

>[!IMPORTANT]
>
>TTL が適用されると、選択した TTL 日数より古いデータが適用されます **永久に** 削除されたため、復元できません。

TTL を適用する前に、TTL の境界内にあるセグメントのルックバックウィンドウを維持する必要があります。 そうしないと、TTL が適用された後に、セグメントの結果が正しくなくなることがあります。 例えば、Adobe Analyticsのデータに 30 日の TTL を適用し、店内トランザクションのデータに 365 日の TTL を適用した場合、次のセグメントは誤った結果を作成します。

* 過去 60 日間に製品ページを閲覧し、その後に店舗での購入が続いた。
* 過去 60 日間に買い物かごに追加し、購入がなかった場合。

逆に、次のような場合でも、正しい結果が得られます。

* 過去 14 日間に製品ページを閲覧し、その後に店舗での購入が続いた。
* 過去 30 日間に、特定のヘルプページがオンラインで表示された。
* 過去 120 日間に製品をオフラインで購入した。
* 過去 14 日間に買い物かごに追加され、購入が続いた。

>[!TIP]
>
>利便性を考慮して、すべてのデータセットに対して同じ TTL を保持できるので、セグメント化ロジックのデータセット間での TTL の影響を心配する必要がありません。

TTL をプロファイルデータに適用する方法について詳しくは、 [プロファイルサービス TTL](../../profile/apply-ttl.md).
