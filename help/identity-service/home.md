# Adobe Experience Platform ID サービス

関連するデジタルエクスペリエンスを提供するには、顧客ベースを構成する現実世界のエンティティの包括的かつ正確な表現が必要です。

今日、組織や企業は、大量の異なるデータセットに直面しています。個々の顧客は、様々な異なる識別子で表されます。 お客様は、様々な web ブラウザー（Safari、Google Chrome）、ハードウェアデバイス（電話、ラップトップ）、その他のユーザー識別子（CRMID、メールアカウント）にリンクできます。 これにより、顧客に関する切り離されたビューが作成されます。

Adobe Experience Platform ID サービスを使用すると、これらの課題を解決し、次の機能を実現できます。

* 異なる ID をリンクする **ID グラフ** を生成することで、顧客が様々なチャネルでブランドとどのようにやり取りするかを視覚的に表現します。
* リアルタイム顧客プロファイルのグラフを作成します。このグラフを使用して、属性と行動を結合することで顧客の包括的なビューを作成します。
* 様々なツールを使用して、検証とデバッグを実行します。

このドキュメントでは、ID サービスの概要と、Experience Platformのコンテキスト内で ID サービスの機能を使用する方法について説明します。

## 用語 {#terminology}

ID サービスの詳細を説明する前に、次の表を参照して主要な用語の概要を確認してください。

| 用語 | 定義 |
| --- | --- |
| ID | ID とは、エンティティに固有のデータです。 通常、これは、個人、ハードウェアデバイス、web ブラウザー（cookie で表される）などの実際のオブジェクトです。 完全修飾 ID は、**ID 名前空間** と **ID 値** の 2 つの要素で構成されます。 |
| ID 名前空間 | ID 名前空間は、特定の ID のコンテキストです。例えば、`Email` の名前空間は、ID 値 **julien<span>@acme.com** と対応できます。 同様に、`Phone` の名前空間は ID 値 `555-555-1234` に対応できます。 詳しくは、[ID 名前空間の概要 &#x200B;](./features/namespaces.md) を参照してください。 |
| ID 値 | ID 値は、実際のエンティティを表す文字列で、名前空間を通じて ID サービス内で分類されます。 例えば、ID 値（文字列） **julien<span>@acme.com** は `Email` 名前空間として分類できます。 |
| ID タイプ | ID タイプは、ID 名前空間のコンポーネントです。 ID タイプは、ID データが ID グラフ内でリンクされているかどうかを指定します。 |
| リンク | リンクまたはリンケージは、2 つの異なる ID が同じエンティティを表すことを確立する方法です。 例えば、「`Email` = julien<span>@acme.com」と「`Phone` = 555-555-1234」の間のリンクは、両方の ID が同じエンティティを表すことを意味します。 これは、julien<span>@acme.comのメールアドレスと 555-555-1234 の電話番号の両方でブランドを操作した顧客が同じであることを示唆しています。 |
| ID サービス | ID サービスはExperience Platform内のサービスで、ID をリンク（またはリンク解除）して ID グラフを管理します。 |
| ID グラフ | ID グラフは、単一の顧客を表す ID のコレクションです。 詳しくは、[ID グラフビューアの使用 &#x200B;](./features/identity-graph-viewer.md) に関するガイドを参照してください。 |
| リアルタイム顧客プロファイル | リアルタイム顧客プロファイルは、Adobe Experience Platform内の次のようなサービスです。 <ul><li>プロファイルフラグメントを結合し、ID グラフに基づいてプロファイルを作成します。</li><li>プロファイルをセグメント化し、アクティベーション用に宛先に送信できるようにします。</li></ul> |
| プロファイル | プロファイルとは、主体、組織または個人を表したものです。 プロファイルは、次の 4 つの要素で構成されます。 <ul><li>属性：属性は、名前、年齢、性別などの情報を提供します。</li><li>動作：動作は、特定のプロファイルのアクティビティに関する情報を提供します。 例えば、プロファイルの動作によって、特定のプロファイルが「サンダルの検索」か「T シャツの注文」かがわかります。</li><li>ID：結合プロファイルの場合、これは人物に関連付けられているすべての ID の情報を提供します。 ID は、個人（CRMID、メール、電話）、デバイス（IDFA、GAID）、cookie （ECID、AAID）の 3 つのカテゴリに分類できます。</li><li>オーディエンスメンバーシップ：プロファイルが属するグループ（常連ユーザー、カリフォルニア州に住むユーザーなど）</li></ul> |

## ID サービスとは

![Experience Platformでの ID の組み合わせ &#x200B;](./images/identity-service-stitching.png)

B2C コンテキストでは、顧客はビジネスとやり取りし、ブランドとの関係を確立します。 典型的な顧客は、組織のデータインフラストラクチャ内の任意の数のシステムでアクティブになっている可能性があります。 特定の顧客は、e コマース、ロイヤルティ、ヘルプデスクシステム内でアクティブな場合があります。 同じ顧客が、匿名で、または認証済みの手段を介して、様々なデバイスの両方に関与する場合もあります。

次のカスタマージャーニーについて考えてみます。

* Julien は、e コマースの web サイトにアカウントを作成し、過去に商品を注文しました。 Julien は通常、個人のノートパソコンを使用して買い物をし、使用するたびにアカウントにログインします。
* しかし、彼女がサイトを訪問した際に、彼女はタブレットを使用してサンダルを検索します。 このセッションでは、別のデバイスを使用しているので、ログインも注文もしていません。
* この時点で、Julien のアクティビティは、次の 2 つの異なるプロファイルで表されます。
   * 最初のプロファイルは e コマースログイン ID です。 このプロファイルは、ユーザー名とパスワードの組み合わせを使用して、e コマースサイトでセッションを認証する場合に使用されます。 このプロファイルは、クロスデバイス識別子で識別されます。
   * 2 つ目のプロファイルはタブレットデバイスです。 このプロファイルは、アカウントにログインせずにタブレットを使用して e コマースサイトを匿名で閲覧した後に作成されました。 このプロファイルは、cookie 識別子で識別されます。
* その後、ジュリアンはタブレットセッションを再開する。 しかし、今回は彼女は自分のアカウントにログインします。 その結果、ID サービスは、Julien のタブレットデバイスのアクティビティと e コマースログイン ID を関連付けるようになりました。
* 今後、ターゲットコンテンツには、Julien の完全なプロファイル、購入履歴、匿名の閲覧アクティビティが反映される可能性があります。

>[!IMPORTANT]
>
>ID サービスを使用して ID をリンクし、異なるシステムに分散している可能性のある顧客の全体像をまとめることができます。

## ID サービスの機能

ID サービスは、その使命を達成するために次の操作を提供します。

* 組織のニーズに合わせてカスタム名前空間を作成します。
* ID グラフを作成、更新、表示します。
* データセットに基づいて ID を削除します。
* ID を削除して、規制への準拠を確保します。

## ID サービスによる ID のリンク方法

>[!IMPORTANT]
>
>ID サービスでは大文字と小文字が区別されます。 例えば、**abc<span>@gmail.com** と **ABC<span>@GMAIL.COM** は、2 つの異なるメール ID として扱われます。

ID 名前空間と ID 値が一致すると、2 つの ID 間のリンクが確立されます。

一般的なログインイベント **2 つの ID を送信** のExperience Platformへの送信：

* 認証済みユーザーを表す人物識別子（CRMID など）。
* Web ブラウザーを表すブラウザー識別子（ECID など）。

次の例をご覧ください。


* ノートパソコンを使用して、ユーザー名とパスワードの組み合わせを使用して e コマース web サイトにログインします。 このイベントは、ユーザーを認証済みユーザーとして認定するので、ID サービスは CRMID を認識します。
* ブラウザーを使用して e コマース web サイトにアクセスした場合も、ID サービスによってイベントとして認識されます。 このイベントは、ECID を使用して ID サービスで表されます。
* ID サービスは、バックグラウンドで 2 つのイベントを `CRM_ID:ABC, ECID:123` として処理します。
   * CRMID:ABC は名前空間であり、認証済みユーザーを表す値です。
   * ECID:123 は、名前空間と、ラップトップでの web ブラウザーの使用状況を表す値です。
* 次に、同じ e コマース web サイトに同じ資格情報でログインしたが、ラップトップの web ブラウザーではなく、電話の web ブラウザーを使用している場合、新しい ECID が ID サービスに登録されます。
* ID サービスは、バックグラウンドで、この新しいイベントを `{CRM_ID:ABC, ECID:456}` として処理します。ここで、CRM_ID: ABC は認証済みの顧客 ID を表し、ECID:456 はモバイルデバイス上の web ブラウザーを表します。

上記のシナリオを考慮すると、ID サービスは `{CRM_ID:ABC, ECID:123}` と `{CRM_ID:ABC, ECID:456}` の間にリンクを確立します。 これにより、3 つの ID を「所有」する ID グラフが作成されます。1 つはユーザー識別子（CRMID）用、2 つは Cookie 識別子（ECID）用です。

詳しくは、[ID サービスによる ID のリンク方法 &#x200B;](./features/identity-linking-logic.md) に関するガイドを参照してください。

## Experience Platform インフラストラクチャ内での ID サービスの役割について

ID サービスは、Experience Platform内で重要な役割を果たします。 これらの主要な統合の一部を次に示します。

* [&#x200B; スキーマ &#x200B;](../xdm/home.md)：特定のスキーマ内で、ID としてマークされたスキーマフィールドを使用すると、ID グラフを作成できます。
* [&#x200B; データセット &#x200B;](../catalog/datasets/overview.md)：データセットがリアルタイム顧客プロファイルへの取り込みに対して有効になっている場合、データセットが ID としてマークされた少なくとも 2 つのフィールドとして指定されていると、ID グラフがデータセットから生成されます。
* [&#x200B; データ収集 &#x200B;](/help/collection/home.md)：データ収集ライブラリ（Web SDKなど）は、エクスペリエンスイベントをAdobe Experience Platformに送信します。 ID サービスは、イベントに 2 つ以上の ID が存在する場合にグラフを生成します。
* [&#x200B; リアルタイム顧客プロファイル &#x200B;](../profile/home.md)：特定のプロファイルの属性とイベントを結合する前に、リアルタイム顧客プロファイルは ID グラフを参照できます。 詳しくは、[ID サービスとリアルタイム顧客プロファイルの関係について &#x200B;](./identity-and-profile.md) に関するガイドを参照してください。
* [&#x200B; 宛先 &#x200B;](../destinations/home.md)：宛先は、ID 名前空間（ハッシュ化されたメールなど）に基づいて、プロファイル情報を他のシステムに送信できます。
* [&#x200B; セグメントの一致 &#x200B;](../segmentation/ui/segment-match/overview.md)：セグメントの一致は、同じ ID 名前空間と ID 値を持つ 2 つの異なるサンドボックスで、2 つのプロファイルを一致させます。
* [Privacy Service](../privacy-service/home.md)：削除リクエストに `identity` が含まれている場合は、Privacy Serviceのプライバシーリクエスト処理機能を使用して、指定された名前空間と ID 値の組み合わせを ID サービスから削除できます。
