# IDサービスの概要

関連するデジタルエクスペリエンスを提供するには、顧客に関する完全な理解が必要です。 これは、お客様のデータが異なる複数のシステムに断片化されている場合に、より難しくなり、個々のお客様が複数の「アイデンティティ」を持っているように見える原因となります。 Adobe Experience Platform Identity Serviceは、デバイスやシステム間でIDをつなぐことで、顧客とその行動をより良く表示できるようにし、効果的な個人のデジタルエクスペリエンスをリアルタイムで提供します。

## IDサービスについて

毎日、顧客は貴社のビジネスとやり取りし、自社ブランドとの関係を絶え間なく深めていきます。 eコマース、忠誠度、ヘルプデスクシステムなど、組織のデータインフラストラクチャ内の任意の数のシステムで、一般的なお客様がアクティブになる場合があります。 同じ顧客が任意の数のデバイスに匿名で関与する場合もあります。 アイデンティティサービスを使用すると、お客様の全体像をまとめ、異なる複数のシステムに分類される可能性のある関連データを集計できます。

消費者とブランドとの関係が日常的に示される例を考えてみましょう。

Maryはeコマースサイトで、過去に数件の注文を完了したアカウントを持っています。 彼女は自分のパソコンを買い物に使い、毎回ログインします。 ただし、いずれかの訪問中、彼女はタブレットを使ってサンダルの買い物をしますが、注文を行わず、ログインしません。

この時点で、Maryのアクティビティは2つの異なるプロファイルとして表示されます。 eコマースログインとタブレットデバイス（デバイスIDで識別される場合があります）。

Maryは後でタブレットセッションを再開し、ニュースレターの購読中に電子メールアドレスを提供します。 これにより、ストリーミング取り込みでは、新しいIDがプロファイル内のレコードデータとして追加されます。 その結果、IDサービスはMaryのタブレットデバイスアクティビティをeコマースアカウントの履歴と関連付けるようになりました。

タブレットを次にクリックすると、ターゲットコンテンツは、未知の買い物客が使用するタブレットだけでなく、Maryの完全なプロファイルと履歴を反映したものになります。

IDサービスが定義および保守するID関係は、リアルタイム顧客プロファイルが顧客の全体像と自社ブランドとのインタラクションを作成するために利用します。 詳しくは、 [リアルタイム顧客プロファイルの概要を参照してください](../profile/home.md)。

### ID

IDとは、エンティティ（通常は個人）に固有のデータです。 ログインID、ECID、忠誠度IDなどのIDは、 **既知のIDと呼ばれます**。

電子メールアドレスや電話番号などのPIIは、顧客を直接識別するために使用します。 その結果、PIIを使用して、システム間で顧客の複数のIDを照合します。

**未知のIDまたは匿名ID** は、実際に使用している人を特定せずにデバイスを取り出します。 このカテゴリには、訪問者のIPアドレスやcookie IDなどの情報が含まれます。 行動データは不明なIDを使用してデバイスから収集できますが、デバイスやメディア間での関連付けは、顧客がその遍歴でPIIを提供するまで制限されます。

次の画像に示すように、既知のIDと匿名IDは共に [IDグラフの重要なコンポーネントで](#identity-graphs)、このドキュメントで後述します。

![プラットフォームでのIDの切り替え](./images/identity-service-stitching.png)

IDサービスの実装の例を次に示します。

- 通信会社は「電話番号」の値に依存する場合があります。電話番号は、オフラインとオンラインの両方のデータセットに対して、同じ関心を持つ人を指します。
- 有料会社は、匿名訪問者の割合が高いので、オフラインデータセットとオンラインデータセットのECIDに「電子メールアドレス」を使用する場合があります。
- 銀行では、支店トランザクションなど、オフラインデータセットで「口座番号」を好む場合があります。 ほとんどの訪問者は訪問中に認証されるので、オンラインデータセットの「ログインID」に依存する場合があります。
- また、GUIDや他の汎用的に一意の識別子など、お客様固有の独自IDを持つ場合もあります。

## ID名前空間とIDのグラフ

顧客は、オンラインとオフラインのチャネルを組み合わせてブランドとやり取りするので、その結果、断片化されたインタラクションを1つの顧客IDにどのように調整するかという課題が生じます。

エクスペリエンスプラットフォームでは、次の2つの概念を通じてこの課題に対処します。 [ID名前空間](#identity-namespaces) と [IDのグラフ](#identity-graphs)。

### ID 名前空間

顧客がWeb、モバイルアプリケーション、コールセンター、ストアフロントなど複数のチャネルで自社ブランドとやり取りしている場合、チャネル間でのアクティビティを観察し追跡できないと、顧客の理解と提供が難しくなります。

各チャネルでの顧客の認識により、複数のデバイスやチャネル開始にわたって顧客を把握する。 これは、ID名前空間を使用することで、Adobe Experience Platformが実現します。
ID名前空間は、データの送信元のコンテキストを提供するために使用される、デバイスIDや電子メールIDなどの識別子です。 ID名前空間は、個々のIDを検索またはリンクし、データの競合を防ぐためのID値のコンテキストを提供するために使用されます。 例えば、ID &quot;123456&quot;はeCommerceシステム内の1人、およびヘルプデスクシステム内の別の人を参照する場合があります。 詳しくは、「 [ID名前空間の概要](./namespaces.md)」を参照してください。

### ID グラフ

ID グラフは、異なる ID 名前空間間の関係のマップで、顧客が様々なチャネルを通じてブランドとどのようにやり取りするかを視覚的に示します。

すべての顧客 ID グラフは、顧客の行動に応じて、ほぼリアルタイムで ID サービスによって管理および更新されます。

ID サービスは、組織のみが表示できるデータに基づいて構築された ID グラフ（「プライベートグラフ」）を管理します。取り込んだデータレコードに複数の ID が含まれている場合、ID サービスはプライベートグラフを増補し、見つかった ID 間の関係を追加します。

IDデータの供給やラベル付けを行う際に考慮すべき潜在的な要因のタイプの例として、「勤務先電話」などの電話番号を使用すると、IDグラフに記載する意図したよりも多くの関係が生じる可能性があります。 多くの従業員が同じ職場番号を参照し、「ホーム」と「モバイル」は、できるだけ正確な関係を保つために役立ちます。

## IDデータのIDサービスへの提供

この節では、Adobe Experience Platformに提供されるデータが、Identity Serviceで顧客ごとにIDグラフを作成する際に使用される前に、どのように処理されるかについて説明します。

### IDフィールドの決定

企業のデータ収集方法に応じて、IDとしてラベルを付けるデータフィールドによって、IDマップに含めるデータが決まります。 Adobe Experience Platformの最大限のメリットと、可能な限り包括的な顧客IDを得るには、オンラインとオフラインの両方のデータをアップロードする必要があります。

- オンラインデータとは、ユーザー名や電子メールアドレスなど、オンラインでの存在と動作を説明するデータです。

- オフラインデータとは、CRMシステムのIDなど、オンラインプレゼンスに直接関係のないデータを指します。 このタイプのデータは、IDをより堅牢にし、異なるシステム間でのデータの統一をサポートします。

### 追加のID名前空間の作成

エクスペリエンスプラットフォームは様々な標準名前空間をオファーしますが、IDを適切に分類するために、追加の名前空間を作成する必要がある場合があります。 詳しくは、「ID名前空間の概要 [」の「組織の名前空間の](./namespaces.md) 表示と作成」の節を参照してください。

>[!NOTE] ID名前空間はIDの修飾子です。 その結果、名前空間を一度作成すると、削除できません。

### エクスペリエンスデータモデル(XDM)にIDデータを含める

プラットフォームが顧客データを編成する標準化されたフレームワークとして、Experience Data Model(XDM)を使用すると、Experience Platformと他のサービス間でデータを共有し、プラットフォームとやり取りすることができます。 詳しくは、 [XDMシステムの概要](../xdm/home.md)。

レコードと時系列の両方のスキーマは、IDデータを含める手段を提供します。 データが取り込まれると、同一性グラフは、共通の同一性データを共有していると見なされた場合に、異なる名前空間のデータフラグメント間に新しい関係を作成します。

### XDMフィールドをIDとしてマーク

レコードまたは時系列のXDMクラス `string` を実装するスキーマ内のタイプのフィールドは、IDフィールドとしてラベル付けできます。 その結果、そのフィールドに取り込まれるすべてのデータは、同一性データと見なされます。

IDフィールドは、共通のPIIデータを共有している場合にIDをリンクすることもできます。
例えば、電話番号フィールドをIDフィールドとしてラベル付けすると、同じ電話番号を使用している他の個人との関係が自動的にグラフ化されます。

>[!NOTE] 結果のIDの名前空間は、フィールドにラベルが付けられた時点で提供されます。

### IDサービスのデータセットの設定

ストリーミング取り込み処理中、アイデンティティサービスは、レコードと時系列データからアイデンティティデータを自動的に抽出します。 ただし、データを取り込む前に、IDサービスを有効にする必要があります。 詳しくは、APIを使用したリアルタイム顧客プロファイルおよびIDサービスのデータセットの [設定に関するチュートリアルを参照してください](../profile/tutorials/dataset-configuration.md) 。

### データをIDサービスに取り込む

IDサービスは、バッチ取り込み [またはストリーミング取り込みによってExperience Platformに送信されるXDM準拠のデータを使用し](../ingestion/batch-ingestion/overview.md)[](../ingestion/streaming-ingestion/overview.md)ます。

## データ管理

Adobe Experience Platformは、プライバシーを考慮して構築され、顧客のPIIデータを保護するためのデータ管理フレームワークが含まれています。 「email」または「phone」名前空間のIDデータはデフォルトで暗号化されますが、機密データが保持される前に確実に暗号化されるように、データ使用ラベルをデータの取り込み時またはプラットフォームに到着後に適用できます。 詳細については、 [データガバナンスの概要を参照してください](../data-governance/home.md)。
